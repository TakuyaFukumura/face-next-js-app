# リアルタイム顔認識・属性推定Webアプリ 仕様書

## 0. 目的と前提

本書は、Next.js を用いた「リアルタイム顔認識・属性推定Webアプリケーション」の実装前に、要求仕様とシステム設計の整理を目的として作成する。

- 対象アプリは **単一画面の SPA 構成** とする
- カメラ映像の解析は **ブラウザ上で完結** させる
- **データベースは使用しない**
- 解析結果は保存せず、セッション中の一時表示のみに限定する
- デプロイ先は Vercel を想定する

## 1. 採用技術の推奨方針

### 1.1 顔認識ライブラリの推奨

本要件では **face-api.js の採用を第一候補** とする。

#### 推奨理由

- 顔検出、表情推定、年齢推定、性別推定を同一ライブラリ系で扱いやすい
- `<video>` と `<canvas>` を組み合わせたブラウザ完結の実装事例が多い
- MediaPipe は顔検出や顔ランドマークには強いが、**少なくとも 2025 年初頭時点で一般的な構成では年齢・性別推定を単独で完結しにくい** ため、別モデルの追加設計が必要になりやすい
- 「まず動くものを実装したい」という観点では、要件充足までの実装負荷を抑えやすい

#### 補足

- face-api.js はモデルファイル容量が比較的大きく、初回ロード時間に注意が必要
- 将来的に高精度化やモバイル最適化を重視する場合は、MediaPipe + 独自属性推定モデルの再検討余地がある

## 2. 機能要件

### 2.1 画面仕様

- 画面は 1 ページ構成とする
- 主な表示要素は以下とする
  - アプリタイトル
  - カメラ映像表示エリア
  - 映像上に重ねる `<canvas>`
  - ステータスメッセージ表示領域
  - カメラ権限未許可時の案内文
  - モデル読み込み中のローディング表示
- PC / スマホの両方で利用可能なレスポンシブ UI とする
- 映像コンテナ内で video と canvas の表示サイズを一致させる

### 2.2 カメラ制御要件

- 初回表示時に `navigator.mediaDevices.getUserMedia()` を用いてカメラ利用を要求する
- 利用許可時は、取得した `MediaStream` を `<video>` に接続して自動再生する
- モバイルブラウザを考慮し、以下を設定する
  - `playsInline`
  - `muted`
  - `autoPlay`
- 可能であれば `facingMode: "user"` を優先し、インカメラを既定値とする
- 権限拒否時は、再試行方法を含むエラーメッセージを表示する
- カメラ未対応ブラウザでは、非対応メッセージを表示する
- ページ離脱時またはコンポーネント破棄時には `MediaStreamTrack.stop()` を実行し、カメラを停止する

### 2.3 リアルタイム解析要件

- 解析開始条件
  - カメラ映像の再生開始後
  - 顔認識モデルのロード完了後
- 解析対象
  - 映像フレーム内の人物の顔
- 解析内容
  - 顔バウンディングボックス
  - 表情推定（例: happy / sad / angry / fearful / disgusted / surprised / neutral）
  - 推定年齢
  - 推定性別
- 表示方針
  - 感情は最も確率の高いラベルを主表示とし、必要に応じてスコアを併記する
  - 年齢は四捨五入した整数、または小数 1 桁で表示する
  - 性別はライブラリ出力に基づく推定値を表示する
- 更新頻度
  - 毎フレーム必須ではなく、**200ms〜500ms 間隔程度** の推論ループを基本とする
  - 端末性能に応じて描画 FPS と推論頻度を分離する
- 複数人対応
  - 同一フレーム内の複数顔検出を許可する
  - 各顔ごとに枠線と属性ラベルを個別表示する

### 2.4 Canvas オーバーレイ描画要件

- `<video>` の上に絶対配置した `<canvas>` を重ねる
- 顔検出結果に応じて以下を描画する
  - 顔の外接矩形
  - 顔近傍のラベル背景
  - 表情 / 年齢 / 性別のテキスト
- video の実寸と CSS 表示サイズの差異を吸収し、座標変換を行う
- リサイズ時は canvas サイズを再同期する
- 前フレームの描画は `clearRect` で消去してから再描画する

### 2.5 状態表示要件

- 少なくとも以下の状態を UI 上で識別可能にする
  - 初期化前
  - モデル読み込み中
  - カメラ起動中
  - 解析中
  - カメラ権限拒否
  - モデル読み込み失敗
  - カメラ取得失敗
  - 顔未検出
- 顔未検出時でも映像は継続表示し、エラー扱いにはしない

### 2.6 データ取り扱い要件

- 映像データ、推論結果、静止画をサーバー送信しない
- ローカルストレージ、IndexedDB、DB への永続保存を行わない
- ログ出力に個人情報相当の推論結果を残さない

## 3. 非機能要件

### 3.1 パフォーマンス

- 初回モデルロード完了の目安: `TinyFaceDetector` + 表情推定 + 年齢/性別推定モデルを読み込む前提で、良好な通信環境では 3〜10 秒以内を目標とする
- 通常動作時の体感更新頻度: 2〜5 FPS 以上の推論更新を目標とする
- UI 操作やスクロールを阻害しないよう、推論ループは過負荷時に間引き可能とする
- モバイル端末では発熱や電池消費を考慮し、推論頻度の調整余地を持たせる

### 3.2 ブラウザ互換性

- 対応優先ブラウザ
  - Google Chrome 最新版
  - Microsoft Edge 最新版
  - Safari 最新版（iOS / macOS）
- Firefox は動作確認対象候補とするが、優先度は上記より低めとする
- `getUserMedia`, `HTMLVideoElement`, `Canvas API`, `WebAssembly` 利用可否を前提とする

### 3.3 セキュリティ / プライバシー

- すべての解析をクライアントサイドで完結させる
- HTTPS 環境でのみカメラ利用が可能な前提を明記する
- 画面上に「映像・推論結果は保存/送信しない」旨を案内する
- 年齢・性別・感情は**推定値であり誤差を含む**ことを UI あるいは注意書きで明示する
- センシティブな用途（本人確認、採用判断、医療判断など）には使用しない前提を明記する

### 3.4 可用性 / 保守性

- モデルロード、カメラ起動、推論処理を責務分離する
- 将来的なライブラリ差し替えを考慮し、推論ロジックを UI から分離する
- エラー文言は UI コンポーネント側で集約管理できる構成が望ましい

## 4. システム構造 / データフロー

### 4.1 全体構造

1. ユーザーがページへアクセスする
2. クライアントコンポーネントがマウントされる
3. 顔認識モデルを `public` 配下からロードする
4. カメラ利用許可を要求する
5. 取得した `MediaStream` を `<video>` に接続する
6. video の再生開始後、定期的に推論処理を実行する
7. 推論結果を `<canvas>` に描画する
8. 状態変化やエラーを UI に反映する
9. ページ離脱時にカメラと推論ループを停止する

### 4.2 データフロー詳細

```text
[Camera Device]
   ↓ getUserMedia
[MediaStream]
   ↓ assign
[video element]
   ↓ frame read
[face-api.js detection / expression / age / gender inference]
   ↓ normalized detection results
[overlay drawing service]
   ↓
[canvas overlay]
   ↓
[User sees bounding boxes and attributes]
```

### 4.3 クライアント内の責務分離イメージ

- `Camera Controller`
  - カメラ起動 / 停止
  - 権限エラー処理
- `Model Loader`
  - モデル読み込み
  - 読み込み状態管理
- `Face Analyzer`
  - 顔検出・表情・年齢・性別推論
  - 推論間隔の制御
- `Overlay Renderer`
  - 座標変換
  - canvas 描画
- `UI State Manager`
  - ローディング、エラー、注意文言の表示

## 6. 実装に向けたステップ（ロードマップ）

### Step 1. 画面の土台作成

- 単一ページ UI を用意する
- video / canvas / status 表示領域を配置する
- レスポンシブレイアウトを整える

### Step 2. カメラ制御実装

- `getUserMedia` によるカメラ起動を実装する
- 権限拒否、未対応ブラウザ、取得失敗のハンドリングを実装する
- アンマウント時の停止処理を実装する

### Step 3. モデル配置とロード処理

- `public/models/face-api/` 配下にモデルを配置する
- クライアント起動後に必要モデルをロードする
- ローディング表示と失敗時メッセージを追加する

### Step 4. 顔解析処理の接続

- 顔検出 + 表情推定 + 年齢推定 + 性別推定を実装する
- 一定間隔の推論ループを構築する
- 顔未検出時の状態更新を実装する

### Step 5. Canvas オーバーレイ描画

- 検出結果に応じた枠線とラベル描画を実装する
- リサイズ時の座標ずれ補正を実装する
- 複数顔表示を確認する

### Step 6. UX / パフォーマンス調整

- ローディング中・解析中・エラー時の文言を改善する
- 推論間隔を調整する
- モバイルでの動作と発熱影響を確認する

### Step 7. クロスブラウザ検証とデプロイ準備

- Chrome / Safari / Edge を中心に実機確認する
- HTTPS 前提のカメラ動作を確認する
- Vercel デプロイ時の静的モデル配信を確認する

## 7. 技術的懸念点・不足情報・提案

### 7.1 モデルファイルの配置方法

- face-api.js のモデルファイルは `public/models/face-api/` に配置し、クライアントから相対 URL で取得する構成が分かりやすい
- Vercel 配信時は静的ファイルとして配信されるため実装しやすい
- ただしモデル容量が大きい場合、初回表示が遅くなるためローディング UI は必須

### 7.2 年齢・性別推定の精度表現

- 年齢・性別は推定結果であり、表示値を断定的に扱わない文言が必要
- 特にデモ用途・学習用途であることを明示したほうが安全

### 7.3 モバイル Safari の注意点

- iPhone / iPad では `playsInline` 未設定だと全画面再生に遷移しやすい
- 自動再生条件や権限挙動に差異があるため、実機確認を早い段階で行うべき

### 7.4 パフォーマンス最適化の優先順位

- まずは精度よりも安定動作を優先し、`TinyFaceDetector` ベースの軽量構成から着手する案が有力
- 推論間隔の調整、表示サイズ制御、モデルの限定ロードで初期段階の負荷を下げられる

### 7.5 現時点で追加確認したい事項

以下は実装着手前に決めておくと手戻りを減らせる。

- 感情表示は「最大スコア 1 件表示」か「上位 3 件表示」か
- 性別ラベル表記を英語にするか日本語にするか
- 年齢表示を整数に丸めるか、小数 1 桁まで出すか
- 複数顔検出時の最大表示人数を制限するか
- モバイル優先か PC 優先か
- 注意書きの強さ（デモ用途、推定誤差、非保存ポリシーの見せ方）

## 8. 実装時の受け入れ基準（案）

- ページ表示時にカメラ権限ダイアログが出る
- 許可後にカメラ映像が表示される
- 顔が映ると枠線が追従する
- 顔ごとに表情 / 年齢 / 性別がオーバーレイ表示される
- 顔が消えると描画が追従して消える
- ページ離脱時にカメラが停止する
- 映像・推論結果が保存・送信されない
- 権限拒否やモデルロード失敗時にユーザーが理解できるメッセージが表示される

以上を、初回実装に向けた仕様・設計のたたき台とする。
